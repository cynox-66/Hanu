# Design System

Version: 1.0

Status: Draft

Owner: Dev Jaiswal

---

# Purpose

The Hanu Design System defines the visual and interaction language of the application.

It ensures every screen feels consistent,
predictable,
calm,
and effortless.

This document prioritizes usability over aesthetics.

---

# Design Philosophy

Hanu is not social media.

Hanu is not an ERP.

Hanu is not enterprise software.

Hanu should feel like:

- Calm
- Friendly
- Trustworthy
- Lightweight
- Familiar

The interface should disappear into the workflow.

The user should focus on their business,
not the software.

---

# Core Design Principles

## Calm

No unnecessary colors.

No flashing animations.

No visual noise.

No clutter.

---

## Large Touch Targets

Every primary interaction should be comfortable with one thumb.

Minimum touch target:

48dp

Preferred:

56dp

---

## One Primary Action

Every screen should have only one obvious action.

The user should never ask:

"What do I press?"

---

## Reduce Reading

Prefer icons.

Prefer photos.

Prefer visual hierarchy.

Avoid paragraphs.

---

## Recognition Over Memory

The app should remind the user.

The app should never expect the user to remember.

---

# Color Philosophy

Colors communicate meaning.

Never decoration.

---

## Primary

Used only for:

Primary Buttons

FAB

Active Navigation

Important Links

---

## Success

Used for

Paid

Delivered

Completed

Success Messages

---

## Warning

Used for

Low Stock

Pending

Unsaved Changes

---

## Error

Used for

Cancelled

Delete

Failures

Invalid Input

---

## Neutral

Everything else.

Backgrounds.

Cards.

Dividers.

Secondary text.

---

# Color Usage Rules

Never use more than

One primary color

One success color

One warning color

One error color

on a single screen.

---

# Typography

Maximum of three sizes.

Large

Page titles

Medium

Section titles

Normal

Everything else

Avoid tiny text.

---

# Font Weight

Bold

Titles

Medium

Buttons

Regular

Body

Never use thin fonts.

---

# Icons

Icons should be simple.

Filled icons preferred.

Never decorative.

Icons must always support text.

Never icon-only navigation.

---

# Buttons

Three button types.

Primary

Filled

Large

One per screen.

---

Secondary

Outlined

Supporting actions.

---

Text Button

Low priority actions.

---

Never place two primary buttons on one screen.

---

# Cards

Cards are the primary information container.

Every card should answer one question.

Example

Order Card

Who?

What?

Status?

Nothing more.

---

# Images

Products are image-first.

Images are always larger than text.

Products should be recognizable before readable.

---

# Lists

Avoid tables.

Prefer cards.

Cards work better on phones.

---

# Forms

Forms should be short.

Maximum

Five inputs

before saving.

Long forms must be split.

---

# Input Rules

Prefer

Dropdown

Toggle

Date Picker

Number Pad

Camera

over typing.

Typing is expensive.

---

# Navigation

Bottom Navigation

Maximum:

Five tabs.

Never six.

---

# Floating Action Button

Always present.

Always obvious.

Always the primary action.

---

# Dialogs

Dialogs ask only important questions.

Delete?

Cancel?

Replace?

Never use dialogs for information.

---

# Empty States

Every empty screen teaches.

Example

No Orders Yet

↓

Your first order will appear here.

↓

Create Order

---

# Loading

Skeleton Loading

Never spinners for long operations.

Never blank pages.

---

# Motion

Motion should communicate.

Never decorate.

Allowed

Fade

Slide

Expand

Not Allowed

Bounce

Spin

Flash

Elastic

---

# Feedback

Every action receives feedback.

Saved

Deleted

Updated

Shared

Failed

The user should never wonder if something worked.

---

# Accessibility

High contrast.

Large buttons.

Large fonts.

Readable spacing.

Touch friendly.

Outdoor visibility.

One handed.

---

# Camera

Camera is a first-class citizen.

Opening the camera should be instant.

Photo review should require one tap.

---

# Photos

Images should be:

Compressed

Optimized

Consistent

Fast

Large enough for products.

---

# Dashboard Philosophy

Dashboard ≠ Analytics

Dashboard = Today's Work

The home screen should answer:

What should I do next?

---

# Screen Density

Every screen should breathe.

Prefer

Whitespace

over

More information.

---

# Error Messages

Bad

Unknown Error

Good

Couldn't save your order.

Your data is safe.

We'll retry automatically.

---

# Confirmation

Only ask for confirmation when:

Deleting

Cancelling

Resetting

Never ask

"Are you sure you want to save?"

---

# Offline Design

Offline should feel normal.

Never panic the user.

Instead say:

Saved locally.

Will sync automatically.

---

# Component Rules

Every reusable component must be:

Predictable

Accessible

Responsive

Reusable

Minimal

---

# Design Checklist

Every screen must answer:

✓ Can my mom understand this?

✓ Can it be used with one hand?

✓ Can it be used outside?

✓ Can it be used quickly?

✓ Does it reduce stress?

✓ Is there only one obvious action?

If any answer is "No",

the design is not ready.

---

# Final Principle

The best interface is the one the user stops noticing.

Hanu should feel less like software,

and more like a helpful habit.
